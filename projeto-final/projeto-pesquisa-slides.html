Otimização de Compilador LLVM para Arquiteturas ULP

Proposta de Pesquisa de Mestrado

Fabricio R. Puente Mansilla

Orientador: Dr. Bruno Cavalcante de Souza Sanches

1. O Desafio da Ultrabaixa Potência (ULP)

1.1. Cenário:

Crescimento de IA embarcada, IoT e Edge Computing exige alto desempenho com consumo mínimo de energia.

1.2. Solução de Hardware:

Arquiteturas Heterogêneas RISC-V (ex: PULP Platform).

  • Paralelismo massivo.
  • Memória local rápida (TCDM/SPM) em vez de caches tradicionais.

1.3. O Problema:

O hardware é eficiente, mas o software (compiladores/runtimes) não acompanha.

2. O Gargalo do OpenMP e LLVM Padrão

2.1. Modelo Atual:

OpenMP é usado para produtividade, mas o task offloading é caro.

2.2. Dados:

O custo de comunicação Host-Cluster pode chegar a 98.000 ciclos de clock.

2.3. Limitação do LLVM:

  • Não realiza Data Tiling automático eficiente para TCDM.
  • Gera transferências excessivas e sincronizações desnecessárias.
  • O offloading pode gastar mais energia que a execução sequencial.

3. Proposta de Solução

3.1. Hipótese:

Passes de otimização no LLVM para Data Tiling e Coerência reduzirão a latência e aumentarão a eficiência energética (Joules/GFLOPs).

3.2. Objetivo Geral:

Desenvolver um Compiler Pass no LLVM que automatize o uso da memória TCDM/SPM.

3.3. A Inovação:

  • Substituir otimização manual por automática.
  • Intervenção direta no pipeline de compilação, sem alterar o código do usuário.

4. Etapas da Pesquisa

4.1. Revisão Sistemática (RSL):

Definir o baseline (LLVM/OpenMP padrão) e benchmarks.

4.2. Design do Artefato:

Implementação dos passes no LLVM para Tiling e ajustes no Runtime OpenMP.

4.3. Experimentação Controlada:

  • Variáveis: Latência de Offload (Ciclos) e Eficiência Energética (Joules/GFLOPs).
  • Comparação: Solução Proposta vs Grupo de Controle (Baseline).

4.4. Análise Estatística:

Uso de Teste T para validar a significância dos resultados.

5. Avançando Além do Estado da Arte

5.1. Atual (AutoDMA/HEROv2):

Funciona bem para alta localidade espacial, mas falha em acessos complexos.

5.2. Diferencial da Proposta:

  • Suporte a Baixa Localidade: Otimização para acessos irregulares (scatter-gather).
  • Hardware-Aware: Uso de extensões de ISA (Zero-Overhead Loops, Post-Increment).
  • Integração: Compatível com ferramentas de deployment de DNNs (DORY/HTVM).

6. Impacto e Contribuições

6.1. Performance:

Atingir 90% do desempenho de código manual (acima da média atual de 85%).

6.2. Produtividade:

Zero alterações no código fonte do usuário (vs. aumento de 2.6x em LOC manual).

6.3. Utilização de FPU:

Manter a unidade de ponto flutuante ocupada (96% - 99%).

6.4. Entregáveis:

  • Pass de Otimização LLVM (Open Source).
  • Artigo em conferência de alto impacto (DAC, ICCAD).

7. Conclusão

  • A pesquisa ataca a tensão entre conveniência do OpenMP e eficiência ULP.
  • Intervenção no compilador é o meio mais eficaz para ganho sistêmico.
